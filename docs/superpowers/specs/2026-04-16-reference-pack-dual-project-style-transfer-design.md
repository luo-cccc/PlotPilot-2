## 背景与目标

### 问题定义

用户希望把一本约 100 万字的参考书输入 PlotPilot，学习其文风、节奏和结构方法，再将这些方法迁移到一部新的原创长篇项目中。

这里的目标不是重写参考书，也不是通过替换专有名词复刻原作，而是把参考书蒸馏为一套可执行的写作约束：

- 文风约束
- 节奏约束
- 结构约束
- 角色功能模板
- 伏笔与转折密度模板

### 本设计的核心目标

1. 支持从整本 `txt/docx/epub` 输入开始。
2. 支持以后重复处理很多本参考书，而不是只服务一次性流程。
3. 先交付最小可跑版，优先保证可重跑、可验证、可扩展。
4. 在风格迁移和版权风险之间建立明确隔离层，避免原文、专名、关键桥段直接流入目标创作工程。

### 本设计的核心结果

系统新增一条 `参考工程 -> Reference Pack -> 目标工程` 的双工程流水线。

其中 `Reference Pack` 是唯一允许从参考工程流向目标工程的中间产物。

## 设计边界

### 要做的事情

- 处理整本参考书的切章、批量导入和分析。
- 复用现有 PlotPilot 章后管线沉淀摘要、三元组、伏笔、时间线、向量索引。
- 输出一个不含原文、仅含抽象规律的 `Reference Pack`。
- 将 `Reference Pack` 应用到原创目标工程，用于试写和后续放量创作。

### 不做的事情

- 不提供“洗稿式重写”能力。
- 不把参考书原文或长摘要直接写入目标工程。
- 不在 MVP 阶段新增复杂前端面板。
- 不在 MVP 阶段改动数据库 schema。
- 不在 MVP 阶段把参考约束直接写进全局 Prompt Plaza。

### 安全与合规边界

本设计默认遵循以下原则：

1. 只迁移风格、节奏、结构方法，不迁移原作可识别内容。
2. `Reference Pack` 只保存抽象特征，不保存可直接复现原作的正文、专名、桥段表。
3. 目标工程中的输出应当“像参考书的创作方法”，而不是“像参考书的剧情内容”。

## 方案比较与结论

### 方案 A：双工程 + Reference Pack 流水线

参考书进入独立的“参考工程”，分析结果蒸馏为 `Reference Pack`，再导入“目标工程”。

优点：

- 污染隔离清晰
- 版权风险最低
- 最适合以后批量处理很多本书
- 便于阶段性重跑和故障恢复

缺点：

- 需要补一组脚本和一个新的中间协议

### 方案 B：单工程内双轨隔离

同一 `novel_id` 或同一工作空间同时保存参考内容和原创内容，通过标签区分。

优点：

- 看起来改动更少

缺点：

- 现有上下文构建、向量检索、知识图谱都容易回捞到参考书内容
- 原创工程和参考工程边界不稳
- 不适合长期批量化

### 方案 C：外部拆书器 + PlotPilot 只负责创作

所有拆书、分析、蒸馏都在外部工具完成，PlotPilot 只接收最后的压缩结果。

优点：

- 对现有服务侵入最小

缺点：

- 流程割裂
- 长期维护会形成两套系统
- 无法充分复用 PlotPilot 现有知识沉淀能力

### 结论

采用 **方案 A：双工程 + Reference Pack 流水线**。

这是最符合本需求的结构：

- 参考书是方法来源
- Reference Pack 是脱敏后的方法载体
- 目标工程是原创输出空间

## 总体架构

### 五阶段总览

1. `参考书预处理`
2. `参考工程入库`
3. `参考包蒸馏`
4. `目标工程落地`
5. `试写校准`

### 架构示意

```text
整本参考书(txt/docx/epub)
    ↓
预处理脚本（切章、清洗、质量检查）
    ↓
标准化章节文件 + manifest + quality_report
    ↓
参考工程导入服务
    ↓
PlotPilot 现有章后管线
    ↓
章节摘要 / 三元组 / 伏笔 / 时间线 / 索引
    ↓
Reference Pack 构建与脱敏
    ↓
reference-pack.json
    ↓
目标工程应用服务
    ↓
style_notes / voice_vault / story_nodes / 局部生成约束
    ↓
试写与校准
```

### 核心原则

#### 工程隔离

- 参考书进入 `reference::<book_slug>`
- 原创项目进入 `target::<project_slug>`
- 两个工程不共享正文，不共享知识检索空间

#### 中间件隔离

- `Reference Pack` 是唯一合法桥梁
- 任何直接从参考工程向目标工程迁移原文的路径都不允许存在

#### MVP 优先级

- 先做 CLI 流水线
- 先复用现有服务
- 先保证可重跑和安全
- 再逐步做 UI 和全局调度

## Reference Pack 边界

### 定位

`Reference Pack` 不是参考书备份，也不是摘要存档，而是参考书的“方法蒸馏包”。

它用于把参考书中可迁移的写作规律带入目标工程。

### 应包含的内容

#### `meta`

- 参考书标题
- 章节数
- 总字数
- 平均章长
- 幕结构统计
- 提取时间
- 提取器版本
- 原始文件 hash

#### `style_profile`

- 叙述视角
- 句式统计
- 对白占比
- 描写密度
- 修辞倾向
- 节奏语气描述
- 可执行风格约束

#### `structure_profile`

- 起承转合比例
- 小钩子频率
- 中转折密度
- 高潮间隔
- 幕平均章数
- 收束速度

#### `character_function_templates`

只保留角色功能位，不保留原角色：

- 主角位
- 搭档位
- 反派压力源
- 信息中介位
- 情感触发位

#### `world_abstractions`

只保留抽象世界模型：

- 权力结构
- 风险等级
- 流动性
- 冲突类型
- 资源稀缺模式

#### `plot_rhythm_profile`

- 信息揭示节奏
- 反转密度
- 伏笔回收窗口
- 支线插入位置
- 高压章节分布

#### `safe_style_samples`

这是面向文风迁移的安全样本，不允许使用参考书原文。

这些样本只保留：

- 句法节奏
- 语气倾向
- 描写组织方式

其语义内容必须换成中性原创表达。

#### `import_map`

用于说明应用到目标工程时如何落地：

- `style_notes`
- `voice_vault`
- `story_nodes`
- 试写脚本的局部生成参数

### 绝对不能包含的内容

- 原书正文
- 原书按章长摘要
- 原角色名、地名、组织名、术语名
- 原书独有设定组合
- 核心桥段顺序表
- 连续高相似表达
- 可直接复现故事的章节事件清单

### 脱敏规则

1. 实体抽象化
2. 剧情规律化
3. 样本内容中性化
4. 专有名词过滤
5. 高相似文本阻断

### 示例结构

```json
{
  "meta": {
    "title": "reference-book",
    "chapter_count": 312,
    "word_count": 1034000
  },
  "style_profile": {
    "narrative_pov": "third_person_limited",
    "dialogue_ratio": 0.34,
    "avg_sentence_length": 18.6,
    "style_rules": [
      "偏短句，转折频繁",
      "情绪描写先动作后心理",
      "高潮段对白密度提升"
    ]
  },
  "structure_profile": {
    "hook_interval_chapters": 2,
    "mid_turn_interval_chapters": 8,
    "major_climax_interval_chapters": 24
  },
  "character_function_templates": [
    {
      "slot": "protagonist",
      "pattern": "被动卷入型成长位"
    }
  ]
}
```

## 最小可跑版流水线

### 阶段 1：整本预处理

输入：

- `txt/docx/epub`

处理：

- 识别目录和章节标题
- 切章
- 去除目录噪声和广告段
- 统计字数
- 标记异常章节

输出：

- `manifest.json`
- `chapters/0001.json ...`
- `quality_report.json`

### 阶段 2：参考工程入库

输入：

- 预处理后的章节目录

处理：

- 自动创建 `reference::<book_slug>`
- 批量写入章节
- 每章导入后调用现有 `ChapterAftermathPipeline`

输出：

- 参考工程章节数据
- 摘要、三元组、伏笔、时间线、向量索引

### 阶段 3：参考包蒸馏

输入：

- 参考工程 `novel_id`

处理：

- 读取章节统计与分析结果
- 计算文风和节奏特征
- 脱敏与抽象化
- 构建 `Reference Pack`

输出：

- `reference-pack.json`
- `safe_style_samples.json`

### 阶段 4：目标工程落地

输入：

- `reference-pack.json`

处理：

- 创建 `target::<project_slug>`
- 写入风格笔记
- 写入安全文风样本
- 初始化结构树参数
- 生成试写用局部约束

输出：

- 一个可直接试写的原创工程

### 阶段 5：试写校准

输入：

- 目标工程
- 试写参数

处理：

- 只生成前 3-10 章
- 跑文风漂移
- 跑章节审阅
- 跑宏观诊断
- 汇总试写报告

输出：

- `pilot_report.json`
- 是否进入放量阶段的建议

## 模块与脚本职责

### 模块层

建议新增目录：

`application/reference_pipeline/`

包含以下服务：

- `book_splitter.py`
- `reference_import_service.py`
- `reference_pack_service.py`
- `reference_redactor.py`
- `reference_apply_service.py`
- `pilot_service.py`

### 各服务职责

#### `book_splitter.py`

- 整本书解析
- 切章
- 文本清洗
- 质量报告生成

#### `reference_import_service.py`

- 创建参考工程
- 批量导入章节
- 逐章触发章后管线
- 记录导入状态

#### `reference_pack_service.py`

- 汇总章节统计、结构统计、风格统计
- 组织 `Reference Pack` 数据结构

#### `reference_redactor.py`

- 专名检测
- 实体抽象化
- 桥段规律化
- 连续高相似表达过滤

#### `reference_apply_service.py`

- 创建目标工程
- 写入 `style_notes`
- 写入安全文风样本
- 初始化结构树参数

#### `pilot_service.py`

- 小规模试写
- 跑诊断
- 生成报告

### CLI 脚本层

建议新增：

- `scripts/reference_pipeline/split_reference_book.py`
- `scripts/reference_pipeline/import_reference_book.py`
- `scripts/reference_pipeline/build_reference_pack.py`
- `scripts/reference_pipeline/apply_reference_pack.py`
- `scripts/reference_pipeline/pilot_generate.py`

### 为什么先做 CLI

因为 MVP 的目标是先把链路跑通，而不是先做交互体验。

CLI 具有以下优势：

- 易于批处理
- 易于失败重跑
- 易于挂到后续调度器
- 不要求先设计新的前端状态管理

## 数据落点与复用策略

### 文件落点

建议所有参考书处理文件落在：

`data/reference_pipeline/<book_slug>/`

建议子目录：

- `prepared/`
- `packs/`
- `reports/`
- `logs/`

### 数据库复用策略

MVP 不改 schema，最大化复用现有库表：

- `novels`
- `chapters`
- `chapter_summaries`
- `triples`
- `story_nodes`
- `voice_vault`
- `voice_fingerprint`

### 目标工程主要写入位置

#### 风格约束

写入 `Bible.style_notes`

原因：

- 已有 UI 可查看
- 已有服务可读
- 与现有世界观/Bible 配置相邻

#### 文风样本

写入 `voice_vault`

原因：

- 可复用现有文风指纹计算机制
- 可复用现有漂移监控逻辑

#### 结构约束

写入 `story_nodes` 初始化参数和结构树骨架

原因：

- 结构树是规划层的现有主载体

### 明确不做的落点

MVP 不把参考约束直接写入全局 Prompt Plaza。

原因：

- Prompt Plaza 当前是全局共享，不是按小说隔离
- 一旦写入全局提示词，容易污染其他工程
- MVP 只需要局部约束能力，不需要全局提示词重构

## 失败恢复与质量闸门

### 失败恢复原则

每个阶段都必须：

1. 产出稳定文件
2. 可单独重跑
3. 不依赖全链路回滚

### 阶段性恢复策略

#### `split`

- 以原文件 hash + 配置作为输出定位键
- 失败后只重跑切章阶段

#### `import`

- 按章幂等
- 允许跳过已成功章节，仅补缺失章节

#### `build pack`

- 失败后不动参考工程
- 只重跑蒸馏和脱敏

#### `apply`

- 失败后不污染既有目标工程
- 默认创建新的目标 `novel_id`

#### `pilot`

- 保留生成内容和诊断报告
- 便于人工判定问题属于风格、结构还是脱敏不足

### 四道质量闸门

#### 切章闸门

- 空章率必须低于阈值
- 极短章、极长章必须出现在 `quality_report`

#### 导入闸门

- 导入章节数与预处理章节数基本一致
- 导入总字数与预处理结果基本一致
- 章后管线成功率达标

#### 参考包闸门

- 脱敏扫描通过
- 不含专名
- 不含长剧情复述
- 不含连续高相似表达

#### 试写闸门

- 文风漂移不连续报警
- 审稿结果达标
- 宏观诊断无明显断裂
- 人工判断“像气质，不像剧情”

## 验证标准与扩展路径

### MVP 成功标准

MVP 不追求“自动证明仿写成功”，而是验证三件事：

1. 流程跑通
2. 方法迁移成功
3. 内容隔离成功

#### 流程跑通

从整本文件开始，到目标工程试写结束，整条链路可执行一次。

#### 方法迁移成功

目标工程在以下方面体现参考书特征：

- 句法节奏
- 对白与描写比例
- 幕长分布
- 钩子和转折密度

#### 内容隔离成功

目标工程中不应出现：

- 参考正文
- 专有名词
- 核心桥段复写
- 可识别的原作事件链

### 批量扩展策略

由于目标场景是以后反复处理很多本书，MVP 需要为批量化预留结构：

- 一个书一个工作目录
- 一个书一个 `Reference Pack`
- 一个书一份 `pilot_report`
- 支持 `split-all`
- 支持 `import-all`
- 支持 `build-pack-all`
- 支持失败书单重跑

### 后续演进方向

MVP 之后的合理扩展顺序如下：

1. 批处理 CLI 聚合入口
2. 按小说隔离的 prompt overlay
3. 参考包版本管理
4. 参考流水线可视化面板
5. 批任务调度与监控

### 最终建议

为了尽快跑通并保持工程稳定，本设计建议坚持以下顺序：

1. 先用 CLI 跑通
2. 先复用现有库表
3. 先做文件化 `Reference Pack`
4. 先做局部约束，不做全局 Prompt 改写
5. 先把可重跑和脱敏安全做扎实，再提升自动化精度

---

## 总结

本设计把“参考书学习”明确拆成三个对象：

- `参考工程`：只负责承载参考书和分析结果
- `Reference Pack`：只负责承载可迁移的抽象规律
- `目标工程`：只负责原创生成和校准

这使得系统既能学习参考书的文风和结构方法，又能最大程度避免把原书内容直接带入原创工程。

对于 MVP，本设计选择：

- 双工程隔离
- CLI 先行
- 不改 schema
- 文件化参考包
- 先试写再放量

这是当前约束下最稳、最易扩展、也最适合长期批量使用的实现路径。

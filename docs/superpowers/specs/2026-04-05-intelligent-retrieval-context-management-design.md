## 核心理念

### 信息降噪与精准投喂

**问题：** 传统 AI 写作工具倾向于将"全部家当"（完整世界观、所有人物设定、全部章节历史）一股脑塞给 AI，导致：
- 信息过载，AI "消化不良"
- 生成内容干巴巴，像设定说明文
- 缺乏针对性，无法聚焦当前章节需求

**解决方案：** 构建高度定制化的"单章执行简报"，而非"全部家当"。

### 设计哲学

将 AI 定位为**"场记与剧务"**，而非"联合编剧"：
- AI 负责：信息检索、上下文组装、技术执行
- 作者负责：创意决策、情节走向、风格把控

### 四个核心模块

1. **角色与视点（POV & Characters）**：明确本章视点人物和出场人物
2. **场景与当下背景（Setting & Relevant Lore）**：只给本章会触发的设定
3. **核心引擎（The Engine）**：本章的任务、节拍、冲突
4. **风格与禁忌（Style & Constraints）**：压制 AI 的"AI 味"

## 模块一：场记智能体（前置推理层）

### 职责

场记智能体是整个系统的"大脑"，负责：
1. 分析章节大纲，理解作者意图
2. 识别动作类型（战斗/潜入/对话/情感）
3. 提取显性实体（人名、地名）
4. 推断隐性需求（如"战斗" → 武器/技能，"潜入" → 地形/安保）
5. 输出结构化 JSON，供后续模块使用

### 实现方式

**轻量级 LLM（Haiku/GPT-4o-mini）+ 异步预加载**

**为什么不用本地 NLP 规则引擎？**

小说的核心魅力在于**潜台词（Subtext）**和**具象化描写（Show, Don't Tell）**。

**示例：**
```
大纲："李明盯着王总那只敲击桌面的手，默默把手伸进了风衣口袋。"

NLP 引擎视角：
- 提取实体：李明、王总、手、风衣、口袋
- 判断场景：日常/穿衣

轻量级 LLM 视角：
- 理解：这是"战斗前夕的剑拔弩张"
- 输出：{"action_type": "combat_prep", "required_props": ["李明的隐藏武器"], "emotional_state": "tense"}
```

LLM 的**零样本泛化能力**能够理解隐喻和动作，而规则引擎需要维护庞大且脆弱的规则库。

### 性能优化：异步预加载

**问题：** API 调用延迟（0.5-1.5 秒）会破坏创作心流

**解决方案：防抖触发 + 流式静默更新**

1. **防抖触发（Debounce）：**
   - 不等用户点击"生成正文"才调用场记 LLM
   - 当用户在写大纲时，键盘停止敲击超过 2 秒 → 后台自动启动场记 LLM

2. **流式静默更新：**
   - 用户点击"生成正文"时，场记结果已就绪
   - 直接拿着完整 Prompt 调用主力生成模型
   - **体感延迟：0**

### 输出格式

```json
{
  "characters": ["李明", "王总"],
  "locations": ["废弃工厂"],
  "action_types": ["combat_prep", "stealth"],
  "trigger_keywords": ["武器", "地形", "安保"],
  "emotional_state": "tense",
  "pov": "李明"
}
```

### 双主轴模型

```
前置场记（轻、快、廉价）
    ↓
图谱检索（传统代码）
    ↓
主力执笔（重、深、昂贵）
```

- **前置场记：** 专职做阅读理解，输出结构化 JSON
- **图谱检索：** 根据 JSON 标签，精准查询数据库
- **主力执笔：** 拿着大纲和精准设定，写出文学美感的正文

## 模块二：叙事事件溯源架构（数据层）

### 设计哲学

**小说是事件驱动的，人物成长由事件推动。**

传统方案的问题：
- **Git 版本化：** 网文式的非线性修改（插入章节）会导致版本链崩塌
- **快照模式：** 丧失叙事因果，AI 只知道结果，不知道原因

**解决方案：事件流溯源（Event Sourcing）**

小说本身就是一条线性时间轴，所有设定变更都挂载在"剧情事件"上。

### 数据结构

#### 静态基座（不变属性）

```json
{
  "entity": "李明",
  "type": "主角",
  "gender": "男",
  "core_personality": "坚韧不拔"
}
```

#### 演化事件流（所有变更挂载在事件上）

```json
{
  "chapter": 51,
  "event_id": "e_089",
  "summary": "龙骨山渊吞噬炎魔晶核",
  "mutations": [
    {
      "attribute": "魔法",
      "action": "add",
      "value": "火系"
    }
  ],
  "timestamp": "2026-04-05T10:30:00Z"
}
```

### 时间线查询

**场景：李明前 50 章是水系，51 章觉醒火系**

**生成第 52 章时：**
- 重放事件流 1-51
- 得出："李明现在是水火双修"
- 同时获取最近事件："因吞噬炎魔晶核获得火系"

**回顾第 30 章时：**
- 重放事件流 1-30
- 得出："李明当时只会水系"

**性能：**
- 一部几百万字小说，核心设定变更事件 < 5000 个
- 重放几千个 JSON 事件耗时 < 1ms

### 优势

1. **非线性修改友好：** 插入章节不会破坏版本链
2. **保留因果关系：** AI 知道"为什么"，而非只知道"是什么"
3. **自动生成上下文厚度：** 最近事件可作为生成提示

**示例：**
```
AI 生成第 52 章时，系统提供：
"李明抬起右手，一团火焰升腾而起。这股力量依然透着
龙骨山渊里那只炎魔的狂躁，让他险些无法控制……"
```

这就真正实现了设定的血肉丰满，而不是干巴巴的属性面板。

## 模块三：知识图谱召回（检索层）

### 召回策略

**混合规则筛选 + 向量搜索**

### 第一步：规则筛选（快速缩小候选集）

- **章节范围：** 当前章节 ± 10 章的相关内容
- **出场人物：** 场记提取的人物列表
- **活跃故事线：** 与当前章节范围交集的故事线

### 第二步：图谱遍历（一度关系 + 触发词召回）

#### 一度关系（必带）

- 李明的性格、装备、当前状态
- 废弃工厂的环境、地形

#### 触发词条件召回（选带）

只有当大纲中出现特定触发词时，才去图谱中调取对应设定：

| 触发词 | 召回内容 |
|--------|----------|
| "战斗" | 武器属性、战斗技能 |
| "魔法" | 力量体系规则 |
| "潜入" | 地形死角、安保规则 |
| "交易" | 经济模式、货币设定 |

**未提到的设定一律锁死，不予抓取。**

例：大纲未提到"历史" → 不召回历史背景

### 第三步：向量搜索补充（语义相关）

- 基于大纲 embedding，检索相关过往章节、事件、伏笔
- 限制数量：最多 5 条
- 用于补充"隐性相关"的内容

### 召回示例

**大纲：**
```
"李明被激怒，拔出了腰间的武器，准备与王总决一死战。"
```

**场记分析：**
```json
{
  "action_types": ["combat"],
  "trigger_keywords": ["武器", "战斗技能"]
}
```

**召回内容：**
1. **一度关系（必带）：**
   - 李明：性格（坚韧）、当前状态（愤怒边缘）
   - 王总：性格（傲慢）、当前状态（暴怒）
   - 场景：废弃工厂的地形

2. **触发词召回（选带）：**
   - 李明的武器：【破军剑】属性、技能
   - 李明的战斗技能：【疾风三连斩】

3. **向量搜索（补充）：**
   - 第 23 章：李明与王总的第一次冲突（相关伏笔）

**不召回的内容：**
- 历史背景（大纲未提及）
- 魔法体系（大纲未提及）
- 其他角色的详细设定（未出场）

## 模块四：Token 预算管理（截断层）

### 硬性规则

**设定上下文占比 ≤ 30%**

把最核心的内存留给"本章大纲"和"写作风格要求"。

### 分层预算分配

假设总预算 35K tokens：

| 层级 | 占比 | Token 数 | 内容 |
|------|------|----------|------|
| Layer 1 | 15% | ~5K | 核心上下文 |
| Layer 2 | 55% | ~20K | 智能检索 |
| Layer 3 | 30% | ~10K | 最近上下文 |

#### Layer 1：核心上下文（~5K tokens）

- 小说元数据（标题、作者、类型）
- 当前章节号和大纲
- 情节弧当前张力水平
- 活跃故事线和待完成里程碑
- Bible 时间线笔记（世界内时间参考）

#### Layer 2：智能检索（~20K tokens）

- 出场人物信息（性格、装备、当前状态）
- 场景信息（地点、环境、地形）
- 触发设定（武器、技能、规则）
- 相关过往章节（向量搜索）
- 相关事件和伏笔

#### Layer 3：最近上下文（~10K tokens）

- 最近 3-5 章摘要
- 最近角色活动
- 最近关系变化
- 未解决的伏笔

### 截断策略

**优先级：Layer 1（必须保留）> Layer 2 > Layer 3**

1. 如果 Layer 1 已超出预算 → 只能截断 Layer 1
2. 如果 Layer 1 + Layer 2 在预算内 → 截断 Layer 3
3. 如果 Layer 1 + Layer 2 超出预算 → 截断 Layer 2

**截断时保留完整语义单元（不切断句子）**

### 重要度排序

在截断前，对召回内容进行重要度排序：

1. **大纲直接提到的实体**（最高优先级）
2. **场景隐性实体**（如地形、环境）
3. **世界观规则**（最低优先级）

## 模块五：冲突处理机制

### 5.1 大纲与设定冲突

#### 问题场景

- 设定库：李明是水系法师
- 大纲：李明释放火球术攻击敌人

**冲突：** 大纲与设定矛盾

#### 解决方案：策略 5（大纲绝对意志 + 异步后置审查）

**核心理念：** 作者在写大纲时的"临时起意"或"破除常规"，往往是整部小说最灵光乍现的神来之笔。系统不应该阻断创作心流。

#### 执行期：大纲即法律（绝对服从，绝不加戏）

- 场记检测到冲突（"李明是水系" vs "大纲写火球术"）
- **系统保持沉默**，以大纲为准生成
- 主力模型写出"释放火球术"，但**绝对不主动解释原因**（不写卷轴，也不写血脉）
- **保持留白**，把悬念和最终解释权留在作者手里

#### 校对期：幽灵批注（非侵入式提示）

正文生成完毕后，侧边栏显示非侵入式批注：

```
⚠️ 场记提示：检测到设定偏离

设定库中李明为 [水系]，此处使用了 [火系]。
```

**特点：**
- 不打断生成流程
- 不强制修改
- 只在侧边栏静默提示

#### 结算期：一键收束逻辑闭环

作者读完这一章后，点击侧边栏的警告，系统给出选项：

**A. 正常伏笔**
- 忽略报错（系统知道你在下大棋，不再提示）

**B. 设定进化**
- 一键将"李明掌握火系魔法"更新到全局设定库
- 系统自动创建演化事件：
  ```json
  {
    "chapter": 51,
    "event_id": "e_089",
    "summary": "李明觉醒火系能力",
    "mutations": [{"attribute": "魔法", "action": "add", "value": "火系"}]
  }
  ```

**C. 笔误修正**
- 点击此项，AI 瞬间将这一小段重写为"释放了高压水刃"

### 5.2 隐性身份管理（视点防火墙 + 情绪降维翻译）

#### 问题场景

- 设定：女二号林雪是卧底（第 1 章起就是，但第 100 章才揭晓）
- 第 10 章大纲：林雪与男主日常互动
- 矛盾：
  - 如果 AI **知道**她是卧底 → 可能写出微妙伏笔（如"她的眼神闪过一丝犹豫"）
  - 如果 AI **不知道** → 写出的互动可能过于真诚，缺少暗线张力

#### 核心问题

**大模型藏不住秘密。** 如果直接告诉 AI"林雪是卧底，注意埋伏笔"，AI 极大概率会"用力过猛"：

```
"林雪微笑着递过咖啡，但她转过头时，眼底却闪过了
一丝作为暗影帝国卧底才会有的冷酷。"
```

这种强行加戏会瞬间击穿悬念。

#### 解决方案：方案 E（视点防火墙 + 情绪降维翻译）

**核心理念：** 把生成正文的 AI 当作纯粹的"体验派演员"，绝不给它看超出当前视点的"全知剧本"。

#### 数据结构：双面人设定

```json
{
  "entity": "林雪",
  "public": {
    "personality": "热情洋溢、粗心大意",
    "habits": "喜欢喝甜咖啡"
  },
  "hidden": {
    "identity": "卧底",
    "traits": "极度警惕、对金属声敏感",
    "reveal_chapter": 100
  }
}
```

#### 生成流程

**情况 1：男主 POV（第 10 章）**

**步骤 1：视点防火墙**
- 系统判断本章 POV = 男主
- **绝对不将 "hidden" 信息发给主力 AI**

**步骤 2：场记的情绪降维翻译**
- 场记智能体（知晓一切的导演）分析大纲
- 大纲："林雪与男主在咖啡厅聊起昨晚的安保泄露事件"
- 场记发现：聊安保泄露，会触动林雪的卧底神经

**步骤 3：生成动作指令（而非告知身份）**
- 场记给主力 AI 的系统提示词：
  ```
  "本章林雪的表演要求：使用表层设定（热情）。
  但在男主提及'安保泄露'时，要求你描写一个极细微的、
  反常的生理反应（例如：笑容停滞了半秒，或手指无意识地
  捏紧了杯子），并且严禁在文本中解释这个动作的原因。"
  ```

**步骤 4：主力 AI 输出**
```
"'昨天的安保系统被人动了手脚。'男主搅动着咖啡说道。
林雪正准备去拿糖罐的手在半空中停顿了极为短暂的一瞬，
随后她极其自然地抓起两包黄糖，笑嘻嘻地说：
'是吗？那这也太可怕了吧，查出是谁干的了吗？'"
```

**情况 2：林雪 POV（第 100 章揭晓）**

- 防火墙解除，"hidden" 信息全量下发
- AI 可以名正言顺地写出她的内心算计和警惕

#### 优势

1. **物理级防剧透：** 主力 AI 根本不知道林雪是卧底，从根源上杜绝了 AI 的"表演型人格"剧透
2. **零认知负担：** 作者不需要打任何 #伏笔 标签，也不需要手动控制
3. **冰山理论的完美实现：** 动作写出来了，但原因留给读者去猜

### 6.2 感官锚点共振（一击致命法则）

#### 核心理念

**最高级的恍然大悟，绝不是理性的罗列，而是"感官的瞬间击穿"。**

**反面教材（清单式写作）：**
```
"男主突然恍然大悟！他想起了那天在咖啡厅她停顿的手指，
想起了她对金属声的敏感，想起了她从未喝过那杯加了黄糖的咖啡……"
```

这种写法虽然逻辑闭环，但毫无文学美感，像在念体检报告。

#### 第 100 章揭晓流程

**步骤 1：寻找当下引子**

大纲："林雪在黑暗中拔出枪，对准了男主。"

场记捕捉当下感官锚点：
- 【听觉-金属枪械的咔哒声】
- 【视觉-黑暗中的冷酷神情】

**步骤 2：感官匹配**

场记在"潜台词账本"中搜索匹配的感官锚点：

```
查询：sensory_anchors.auditory LIKE "%金属%"
结果：第 35 章 - "对金属碰撞声极度敏感"
```

**步骤 3：一击致命（只提取一个最契合的伏笔）**

场记给主力 AI 的指令：

```
"在揭晓身份的瞬间，用男主听到的'拔枪的金属咔哒声'，
去共振男主记忆中'第 35 章她听到金属掉落时的反常'。
只需写这一个闪回，严禁罗列其他回忆。"
```

**步骤 4：主力 AI 输出**

```
"黑暗中传来保险栓拉动的咔哒声。极其清脆。
那一瞬间，男主的脑海里不可遏制地闪过几个月前，
那枚硬币掉在铁桌上的声音——当时林雪也是这样，
肌肉在一瞬间绷紧到了极致。他终于明白了。"
```

   - 第 67 章：对某个地点的熟悉
   - 第 78 章：与陌生人的眼神交流
   - 第 89 章：对武器的专业判断

建议在接下来的第 101 章【男主战后独处】或第 102 章

## 模块七：防同化机制（文风金库与指纹对抗）

### 核心问题

**文风漂移（Voice Drift）与 AI 味同化**

在漫长的连载中，即使设定了极度详尽的风格提示词，大模型在海量生成时依然会不可抗拒地产生"均值回归"，悄无声息地滑向那种句式对称、形容词堆砌、四平八稳的"标准 AI 翻译腔"。

### 7.1 动态语料"采血"机制

#### 核心理念

**真正的文风无法用自然语言精确描述，只能从作者的实际修改中"采血"学习。**

#### 触发时机

- 作者手动删改 AI 生成的段落
- 作者重写 AI 生成的对话/描写
- 作者在 AI 生成基础上进行精修

#### 采血流程

```
AI 生成原文 → 作者修改 → 系统捕获差异 → 存入文风金库
```

#### 示例

```
AI 原文："他感到非常愤怒，心中燃起了熊熊怒火。"
作者改为："他把烟头按死在掌心。"

系统记录：
{
  "type": "emotion_expression",
  "ai_version": "直接陈述 + 比喻堆砌",
  "author_version": "动作暗示",
  "lesson": "情绪通过动作展现，避免直接陈述"
}
```

#### 金库结构

```json
{
  "author_id": "user_001",
  "voice_samples": [
    {
      "sample_id": "vs_001",
      "chapter": 5,
      "scene_type": "conflict",
      "ai_original": "他感到非常愤怒，心中燃起了熊熊怒火。",
      "author_refined": "他把烟头按死在掌心。",
      "diff_analysis": {
        "removed": ["非常", "熊熊", "感到"],
        "added": ["按死", "掌心"],
        "pattern": "show_not_tell"
      }
    }
  ],
  "total_samples": 1247,
  "last_updated": "2026-04-05"
}
```

### 7.2 文风指纹提取（Voice Fingerprint）

#### 分析维度

**1. 词法基因**
- 高频动词偏好（如"按死" vs "熄灭"）
- 冷门词汇使用率
- 形容词/副词密度（专业作者通常更低）
- 感叹号/省略号使用频率

**2. 句法节奏（Pacing）**
- 短句/长句比例
- 平均句长
- 句式变化频率（避免单调）
- 段落长度分布

**3. 场景风格偏好**
- 对话场景：对话/叙述比例
- 动作场景：是否偏好短句 + 动词密集
- 情感场景：是否通过环境/动作暗示，而非直接陈述
- 高潮场景：是否切断感官描写，只保留对白

**4. 叙事视角特征**
- 第一人称/第三人称限制视角的语气差异
- 内心独白的表达方式
- 旁白介入的频率和方式

#### 指纹生成

```json
{
  "author_fingerprint": {
    "lexical": {
      "verb_preference": ["按死", "绷紧", "撕裂"],
      "adjective_density": 0.08,  // 低于 AI 平均 0.15
      "exclamation_rate": 0.02    // 极少使用
    },
    "syntax": {
      "avg_sentence_length": 12.3,
      "short_long_ratio": 2.1,    // 短句是长句的 2 倍
      "paragraph_rhythm": "varied" // 长短交替
    },
    "scene_style": {
      "dialogue_ratio": 0.65,     // 对话占 65%
      "action_verb_density": 0.22,
      "emotion_expression": "action_based" // 通过动作表达
    },
    "narrative_voice": {
      "pov_consistency": "strict_limited", // 严格限制视角
      "inner_monologue": "minimal",        // 极少内心独白
      "narrator_intrusion": "none"         // 无旁白介入
    }
  }
}
```

#### 指纹更新策略

- 每 10 次"采血"后，重新计算指纹
- 指纹随创作演进动态调整（允许作者风格自然成长）
- 保留历史指纹版本（如"前期风格" vs "成熟期风格"）

### 7.3 生成端的"抗体"拦截

#### 第一道防线：生成前约束（Prompt 注入）

将文风指纹转化为生成约束：

```
风格要求（最高优先级）：
- 严格执行 Show, Don't Tell：禁止直接陈述情绪（如"他很愤怒"），
  必须通过动作/对话/环境暗示
- 句式节奏：短句与长句比例 2:1，避免连续 3 个以上长句
- 词汇偏好：多用具象动词（如"按死"、"绷紧"），
  少用抽象形容词（如"巨大的"、"强烈的"）
- 对话占比：本章对话应占 60-70%
- 禁用词汇：["毫不夸张地说", "宛如", "不仅...而且...", 
            "熊熊", "滔滔不绝", "如同实质般"]
```

#### 第二道防线：生成后过滤（去 AI 味过滤器）

**检测 AI 高频俗套句式：**

```python
AI_CLICHE_PATTERNS = [
    r"毫不夸张地说",
    r"宛如.*?般",
    r"不仅.*?而且.*?",
    r"如同.*?一般",
    r"仿佛.*?似的",
    r".*?的.*?的.*?的",  # 连续三个"的"
    r"熊熊.*?火",
    r"滔滔不绝",
    r".*?实质.*?实质",   # "宛如实质般的实质"
]
```

**过滤流程：**

```
AI 生成正文 
    ↓
扫描俗套句式
    ↓
检测到 → 标记该段落 → 自动重写（注入"避免此句式"约束）
    ↓
未检测到 → 进行文风指纹匹配
    ↓
匹配度 < 80% → 标记"风格偏离" → 侧边栏提示（非强制）
    ↓
匹配度 ≥ 80% → 通过，呈现给作者
```

**重写示例：**

```
检测到："他感到非常愤怒，心中燃起了熊熊怒火。"
触发规则：["熊熊", "感到", 直接陈述情绪]

自动重写约束：
"重写上一句，要求：
1. 不使用'熊熊'等俗套比喻
2. 不直接陈述'愤怒'
3. 通过具体动作展现情绪
4. 参考作者偏好动词：按死、绷紧、撕裂"

重写结果："他把烟头按死在掌心。"
```

### 7.4 文风漂移监控（Voice Drift Alert）

#### 长期监控指标

- 每 10 章计算一次"文风一致性评分"
- 对比当前章节与"文风指纹"的匹配度
- 如果连续 5 章匹配度 < 75%，触发警告

#### 警告界面

```
⚠️ 文风漂移警告

检测到最近 5 章（第 45-50 章）的文风与您的指纹偏离：
- 形容词密度上升 40%（0.08 → 0.11）
- 短句比例下降 25%（2.1 → 1.6）
- 检测到 12 处 AI 俗套句式

建议操作：
1. 查看偏离片段（已高亮标记）
2. 手动精修 2-3 个典型段落，系统将学习您的修正
3. 重新生成这 5 章（基于更新后的文风约束）
```

### 7.5 文风金库的进阶应用

#### 1. 多风格管理（适用于多 POV 小说）

- 为不同 POV 角色建立独立的文风指纹
- 例：男主 POV（冷峻、短句）vs 女主 POV（细腻、长句）
- 生成时根据当前 POV 自动切换文风约束

#### 2. 风格演化追踪

- 记录作者风格的自然成长
- 例：前 50 章（青涩期）vs 后 50 章（成熟期）
- 允许作者选择"锁定早期风格"或"跟随演化"

#### 3. 风格迁移（实验性）

- 作者可以导入其他作品的文风指纹
- 例："让这一章写得像我的上一部小说"
- 系统融合两个指纹，生成混合风格

## 全局大修工具

### 问题场景

当你完成了 100 章的初稿，决定进行一次**全局视角的"大修（Macro-editing）"**时，比如你想把男主前期的性格从"热血"改成"冷酷"，传统方式需要：
- 逐章阅读几十万字
- 手动标记需要修改的片段
- 逐个重写对话、行为、心理描写
- 检查逻辑合理性

这是一个令人绝望的工作量。

### 解决方案：事件流基因重组（Event-Stream Refactoring）

**核心理念：不改血肉（正文文字），只改骨架（事件日志），让血肉自动重新生长。**

### 步骤 1：逻辑断点扫描（扫描"账本"而非"文本"）

当你在控制面板将男主的底层核心设定从"热血"改为"冷酷"时，系统**绝对不会**去逐字阅读那几十万字的正文。

相反，系统会去遍历底层的"演化事件流"。因为每一个事件都带有元数据（如 `[动机: 冲动]`），系统会瞬间抓取出所有与"冷酷理性"相悖的节点，生成一张**"逻辑断点拓扑图"**。

**示例：**

```
逻辑断点拓扑图：

第 5 章：e_012（拍桌争吵）[动机: 冲动] ⚠️
第 23 章：e_045（冲动救人）[动机: 冲动] ⚠️
第 67 章：e_089（激昂演讲）[动机: 冲动] ⚠️
```

这比逐字扫描文本要精准无数倍，因为它抓取的是**底层叙事逻辑**，而不是表层关键词。

### 步骤 2：靶向修复（对话式协作）

面对这张拓扑图，系统不会自作主张地去改写情节，而是召唤出"场记智能体"，与你进行一场极其高效的"剧本外科手术"。

系统会聚焦核心冲突点，例如第 23 章的救人事件：

```
场记智能体：
"第 23 章事件 e_045（冲动救下流浪汉）与新人设冲突。
如果男主现在是极致理性的，我们需要给这个行为注入一个'功利性动机'。
我建议提取第 15 章设定库里的'黑市纹身'，让他因为认出了
流浪汉身上的纹身才决定出手。这样改可以吗？"

作者：
"改成他需要流浪汉做诱饵"

系统修改事件流：
{
  "event_id": "e_045",
  "summary": "理性救人（需要诱饵）",
  "mutations": [{"动机": "冲动" → "计算"}]
}
```

此时，你只需要点头或者给出一句微调，这个底层事件就被修复了。

### 步骤 3：局部重渲染（只改变相关段落）

当底层事件流的逻辑全部跑通理顺后，最后一步才是动正文。

系统**只抽取出那些被修改了事件节点的相关段落**进行重新渲染。

由于底层属性已经变成了"冷酷"，负责主力生成的 AI 在重写对话时，会自动剥离原本的感叹号和长句，将男主拍桌而起的咆哮，自然而然地重渲染为一句简短、冰冷的反问。

**没有被波及的无辜情节**（比如客观的风景描写、配角之间的独立支线），**连一个标点符号都不会被改变**。

### 优势

将令人绝望的"几十万字文本大修"，变成了一次"几十个核心 JSON 事件的审核与微调"。

## 沙盘试读（验证大修效果）

### 问题

完成事件流修复 + 局部重渲染后，如何快速验证"冷酷男主"版本是否对味？

### 解决方案：角色音轨切片 + 叙事心电图导航

### 步骤 1：极速语感验证（角色音轨切片）

专业作家查验人设，最快、最准的方式不是看剧情，而是听**"台词"和看"关键动作"**。

系统在沙盘模式下，首先不提供完整章节，而是把新版男主的 **50 句核心对白和 20 个关键决策动作**，剥离所有旁白和环境描写，像"剧本围读"一样高密度地连续打在屏幕上。

**示例：**

```
【音轨切片 - 新版男主】

对白 1："是吗。"（第 5 章）
对白 2："你继续。"（第 12 章）
对白 3："不需要。"（第 23 章）
...
动作 1：他眼神冷了下来。（第 5 章）
动作 2：他转身离开，没有回头。（第 23 章）
...
```

如果连看这几十句连续的台词，你能感受到一种扑面而来的"冷酷和生人勿近"，那么新的人设基因就已经成功植入了。

**这比通读几万字正文的效率高一百倍。**

### 步骤 2：宏观排雷导航（叙事心电图）

语感确认无误后，进入排雷阶段。

屏幕上方展现整个 100 章的"情绪与行为曲线"对比：

```
原版（热血男主）：
情绪曲线 ▁▃▅▇█▇▅▃▁▃▅▇█  （剧烈锯齿）

新版（冷酷男主）：
情绪曲线 ▁▁▂▁▁▁▂▁▁▁▁▁  （平缓低谷）
```

这不仅是图表，更是**导航器**。

如果你在这条代表"冷酷"的平缓曲线上，突然看到第 45 章有一个异常高耸的"激昂尖峰"，你就知道：系统的自动修复在这里漏掉了某个残留的"热血"情节。

### 步骤 3：无缝下潜与归档

## 卡文破局工具

### 问题场景

当你面对一个空白文档，连大纲都还没写出来，陷入了严重的"卡文（Writer's Block）"状态。

### 核心问题

卡文（尤其是面对白纸的"首段卡文"）的本质不是"没有想法"，而是**"想法太多且充满矛盾，导致认知过载，大脑启动了自我保护性的宕机"**。

### 解决方案：张力弹弓模型（Tension Slingshot）

**核心理念：** 把作者从"从零创造的上帝"，变成了"收拾烂摊子的修理工"。

人类大脑在"无中生有"时最容易卡壳，但在面对一个"充满戏剧冲突和漏洞的极点画面"时，修补逻辑的本能会被瞬间激发。

### 步骤 1：挂载锚点（纯感性输入）

系统给出的界面极其干净，只有一个输入框，上面写着：

```
"抛弃一切逻辑。描述一个你现在脑海里最具画面感、
最让你肾上腺素飙升或心痛的瞬间。"
```

假设你输入：

```
"大雪天，主角跪在地上，亲手斩下了恩师的头颅。"
```

此时，你不需要知道主角是谁，恩师是谁，为什么杀他。你只需要提供这个感官上的"情绪高点"。

### 步骤 2：拨动弹弓（AI "如果"变异）

接到这个画面后，AI 不会去问你"他为什么要杀"，而是直接去**破坏这个画面的固有常识**，制造极端的叙事张力。

AI 会瞬间吐出 3 个基于该画面的致命"What If"：

**变异 1（倒错张力）：**
```
"如果斩下头颅的那一刻，主角发现面具下的脸，
其实是自己失散多年的亲弟弟，而真正的恩师正站在高塔上鼓掌？"
```

**变异 2（绝望张力）：**
```
"如果恩师的血液是拯救这个世界瘟疫的唯一解药，
但一旦斩下，主角自己就会立刻感染瘟疫倒下？"
```

**变异 3（系统乱入）：**
```
"如果这不是真实世界，而是主角在虚拟潜行中第 1000 次尝试打破死局，
但这第 1000 次，恩师突然脱离了程序设定，对他笑了？"
```

### 步骤 3：释放弹弓（事件流倒推）

当你看到某个"如果"瞬间瞳孔地震，觉得"卧槽，这个有意思"时，认知僵局就被打破了。

你点击**变异 2**。

此时，系统立刻切入场记智能体诊断，利用底层的"事件流架构"，直接往回倒推 3 个必须存在的历史事件骨架：

```
事件 -3：恩师当年为了救主角，主动感染了原始毒株。

事件 -2：世界爆发瘟疫，全人类将恩师视为必须剿灭的怪物。

事件 -1：主角为了保全恩师的名誉，抢在所有人之前接下了猎杀任务。
```

## 中段塌陷诊断与干预

### 问题场景

长篇连载中，还有一种同样致命的卡文叫**"中段塌陷（The Muddy Middle）"**——通常发生在 40 万字到 60 万字之间，世界观已经交代完毕，结局的大高潮还很遥远，剧情开始像流水账一样原地踏步。

### 症状

- 剧情像流水账（日常堆砌，缺乏推进感）
- 角色原地踏步（没有成长或变化）
- 冲突疲软（小打小闹，缺少生死攸关的张力）
- 读者流失率飙升（追读断崖式下跌）

### 解决方案：叙事熵增检测 + 中段高潮注入

#### 叙事熵定义

```
熵值 = 事件的不可预测性 + 后果的不可逆性
```

#### 诊断流程

系统分析 40-60 万字区间的事件流：

```
事件 e_120：日常对话（熵值：2/10）
事件 e_121：小冲突（熵值：3/10）
事件 e_122：日常对话（熵值：2/10）
事件 e_123：日常对话（熵值：2/10）
...
```

如果连续 10 个事件熵值过低（都是"日常对话"、"小冲突"），系统警告：

```
⚠️ 叙事熵衰减警告

检测到第 120-130 章（40-43 万字）叙事熵持续低迷：
- 平均熵值：2.3/10（健康值应为 5-7）
- 连续 10 个事件无重大变化
- 读者疲劳风险：高

建议操作：
注入"高熵事件"以重新激活张力
```

#### 干预建议

**注入"高熵事件"：**

1. **不可逆的选择**
   - 杀人、背叛、告白
   - 一旦发生，无法撤回

2. **不可预测的变量**
   - 新角色登场
   - 新势力介入
   - 世界观扩展（揭示隐藏规则）

3. **中段高潮注入**
   - 在 50 万字处设计"假高潮"或"中场逆转"
   - 逆转模板：
     - 真相揭露（世界观颠覆）
     - 身份反转（敌友互换）
     - 代价显现（前期行为的恶果爆发）

**示例建议：**

```
系统建议：在第 125 章注入高熵事件

方案 1：真相揭露
"揭示男主一直信任的导师，其实是幕后黑手"

方案 2：不可逆选择
"男主必须在救恩师和救爱人之间做出选择，
且选择后另一方必死"

方案 3：新势力介入
"第三方神秘组织突然介入，打破原有势力平衡"
```
